alluvial diagrams in ggplot2

Jason Cory Brunson

2017-04-12

This vignette should

  • define the essential components of alluvial diagrams as used in this package’s naming schemes and documentation (axis, alluvium, stratum, lode, flow),
  • describe the alluvial data structures recognized by ggalluvial,
  • illustrate each of the ggalluvial stats and geoms, and
  • showcase some of the most popular variants on the theme (including a mosaic-like full-width stratum plot, a parallel sets plot, a categorical alluvial diagram, and a weighted bump plot).

Alluvial diagrams

Alluvial data

One row per alluvium

ggalluvial recognizes two “alluvial data” formats. The first follows the visual arrangement of an alluvial diagram: Each row corresponds to a subset or amount of observations that take a specific value at each variable, and each variable has its own column. An additional column contains the weight of each row. This is the format into which the base as.data.frame() function transforms an array, for instance the 4-dimensional UCBAdmissions dataset:

dplyr::tbl_df(as.data.frame(UCBAdmissions))

##       Admit Gender   Dept  Freq
##      <fctr> <fctr> <fctr> <dbl>
## 1  Admitted   Male      A   512
## 2  Rejected   Male      A   313
## 3  Admitted Female      A    89
## 4  Rejected Female      A    19
## 5  Admitted   Male      B   353
## 6  Rejected   Male      B   207
## 7  Admitted Female      B    17
## 8  Rejected Female      B     8
## 9  Admitted   Male      C   120
## 10 Rejected   Male      C   205
## # ... with 14 more rows
is_alluvial(as.data.frame(UCBAdmissions), logical = FALSE)
## Warning in is_alluvial_alluvia(data = data, ..., logical = logical):
## Missing alluvia for some stratum combinations.
## [1] "alluvia"

This form was inherited from the first version of ggalluvial, which was modeled after the alluvial package. It required a stark departure from the usual position aesthetics: The user declares any number of axis variables, which stat_alluvium() and stat_stratum() recognize and process in a consistent way:

ggplot(as.data.frame(UCBAdmissions),
       aes(weight = Freq, axis1 = Gender, axis2 = Dept)) +
  geom_alluvium(aes(fill = Admit)) +
  geom_stratum() + geom_text(stat = "stratum") +
  scale_x_continuous(breaks = 1:2, labels = c("Gender", "Dept")) +
  ggtitle("UC Berkeley admissions and rejections, by sex and department")

An important feature of these diagrams is the meaningfulness of the vertical axis: No gaps are inserted between the strata, so the total height of the diagram reflects the cumulative weight of the observations. The diagrams produced by ggalluvial conform (somewhat; see below) to the “grammar of graphics” principles of ggplot2, and this prevents users from producing “free-floating” diagrams like the Sankey diagrams showcased here. ggalluvial parameters and existing ggplot2 functionality can also produce parallel sets, illustrated here using the Titanic dataset:1

ggplot(as.data.frame(Titanic),
       aes(weight = Freq,
           axis1 = Survived, axis2 = Sex, axis3 = Class)) +
  geom_alluvium(aes(fill = Class),
                width = 1/8, knot.pos = 0) +
  geom_stratum(width = 1/8) + geom_text(stat = "stratum") +
  scale_x_continuous(breaks = 1:3, labels = c("Survived", "Sex", "Class")) +
  coord_flip() +
  ggtitle("Titanic survival by class and sex")

This format and functionality are useful and will be retained in future versions. They also involve some conspicuous deviations from ggplot2 standards:

  • The axis[0-9]* position aesthetics are non-standard.
  • stat_alluvium() ignores any argument to the group aesthetic; instead, StatAlluvium$compute_panel() uses group to link the rows of the internally-transformed dataset that correspond to the same alluvium.
  • The horizontal axis must be manually corrected (using scale_x_continuous()) to reflect the implicit categorical variable identifying the axis.

Furthermore, format aesthetics like fill are necessarily fixed for each alluvium; they cannot, for example, change from axis to axis according to the value taken at each. This means that this format and functionality cannot produce alluvial diagrams with the color schemes featured here (“Alluvial diagram”) and here (“Controlling colors”).

One row per lode

The second “alluvial” format recognized by ggalluvial contains one row per lode, and can be understood as the result of “gathering” (in the dplyr sense) the axis columns of a dataset of the preceding kind (one row per alluvium) into a “key” column indicating the axis and a “value” column indicating the value taken at that axis. The crucial difference is that this format requires an additional indexing column that links the rows corresponding to a common alluvium, as illustrated below using the to_lodes() defaults on the Titanic dataset:

dplyr::tbl_df(to_lodes(as.data.frame(Titanic), axes = 1:4))
## # A tibble: 128 × 4
##     Freq    id    key  value
##    <dbl> <int> <fctr> <fctr>
## 1      0     1  Class    1st
## 2      0     2  Class    2nd
## 3     35     3  Class    3rd
## 4      0     4  Class   Crew
## 5      0     5  Class    1st
## 6      0     6  Class    2nd
## 7     17     7  Class    3rd
## 8      0     8  Class   Crew
## 9    118     9  Class    1st
## 10   154    10  Class    2nd
## # ... with 118 more rows

The same stat and geom can receive this data format using a different set of positional aesthetics, also specific to ggalluvial:

  • x, the “key” variable indicating the axis to which the row corresponds, which are to be arranged along the horizontal axis;
  • stratum, the “value” taken by the axis variable indicated by x; and
  • alluvium, the indexing scheme that links the rows of a single alluvium.

We can illustrate this usage too on the Titanic dataset:

ggplot(to_lodes(as.data.frame(Titanic), axes = 1:4),
       aes(x = key, stratum = value, alluvium = id,
           weight = Freq)) +
  geom_alluvium() +
  geom_stratum() + geom_text(stat = "stratum")

This format and functionality also allow weights to vary from axis to axis, which allows users to produce weighted bump charts like those showcased here. In these cases, the strata and the alluvia are the same, so either stratum or alluvium can be specified while omitting the other. As an example, we can assign each country in the alluvial Refugees dataset a region, allowing us to compare refugee volumes between countries and between regions simultaneously:

data(Refugees, package = "alluvial")
country_regions <- c(
  Afghanistan = "Middle East",
  Burundi = "Central Africa",
  `Congo DRC` = "Central Africa",
  Iraq = "Middle East",
  Myanmar = "Southeast Asia",
  Palestine = "Middle East",
  Somalia = "Horn of Africa",
  Sudan = "Central Africa",
  Syria = "Middle East",
  Vietnam = "Southeast Asia"
)
Refugees$region <- country_regions[Refugees$country]
ggplot(data = Refugees,
       aes(x = year, weight = refugees, stratum = country)) +
  geom_alluvium(aes(fill = country, colour = country), decreasing = FALSE) +
  facet_wrap(~ region, scales = "fixed")
